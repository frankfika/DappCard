# Bonjour Web3 版产品逻辑草案

## 1. 文档目的

这份文档用于帮助定义一个“Bonjour 的 Web3 版本”应该怎么做，重点不是代码实现，而是先把产品逻辑、传播方式、身份体系和最小可行版本想清楚。

当前假设：

- `Bonjour` 的核心价值偏向“轻量社交连接、快速建立关系、带一点身份展示和互动”。
- 目标产品不是传统独立 App，而是一个更容易在 `Telegram`、`Discord`、`X` 等社群里传播的轻量 DApp。
- 产品应具备一定的去中心化身份能力，但不应该一开始就把 DID 做得太重。

如果这个假设和你心里的 `Bonjour` 不完全一致，这份文档也仍然可以作为一个 Web3 社交产品的起点。

## 2. 一句话定义

这是一个面向 Web3 社群传播的轻量社交 DApp：

- 用户通过 `Telegram`、`Discord`、链接卡片、邀请码等方式进入。
- 用户用一个“轻量身份名片”表达自己是谁、来自哪里、在找什么人。
- 用户通过打招呼、匹配、加入频道、完成协作等行为产生连接。
- 用户的社交身份可以逐步升级为一个简单 DID，用于和其他应用交互。

可以先把它理解成：

`Web2 的轻量社交名片/打招呼工具` + `Web3 的可验证身份层` + `社群裂变传播机制`

## 3. 核心问题

做这个产品前，先要回答 4 个问题：

1. 用户为什么愿意创建这个身份？
2. 用户为什么愿意分享出去？
3. 别人为什么愿意点进来？
4. 这个身份为什么值得在其他应用里继续使用？

如果这 4 个问题没有回答清楚，产品就会变成一个“做出来但没人持续使用的链上名片”。

## 4. 推荐定位

### 4.1 最推荐的定位

建议不要一开始做成“泛社交产品”，而是做成：

**基于社群传播的 Web3 身份社交卡片 + 轻互动网络**

也就是：

- 第一层是 `身份展示`
- 第二层是 `连接和打招呼`
- 第三层才是 `DID 和跨应用能力`

原因很简单：

- Web3 用户不愿意一上来填复杂资料。
- DID 对多数用户不是第一需求，而是“有了社交行为之后，身份沉淀出来”。
- `Telegram` 和 `Discord` 的传播更适合“点开即看、轻量注册、可以转发”的产品。

### 4.2 不建议的定位

不建议第一版直接做成：

- 一个完整去中心化社交网络
- 一个重钱包、重链交互的 DApp
- 一个需要强实名或复杂 DID 解析的协议产品
- 一个功能太多的链上 LinkedIn

这些方向太重，传播阻力大，而且不利于冷启动。

## 5. 产品核心价值

这个产品的价值可以分成 3 层。

### 5.1 对用户的价值

- 更快地介绍自己
- 更方便地在社群中建立弱连接
- 更容易被发现和被联系
- 让自己的链上足迹、兴趣和能力被结构化表达

### 5.2 对社群的价值

- 帮助群成员快速认识彼此
- 形成更高质量的连接和匹配
- 给活动、项目、DAO、社区运营提供身份和关系工具

### 5.3 对生态的价值

- 形成可复用的轻 DID
- 让其他应用能够读取用户的社交画像、偏好、成就和关系
- 为后续的推荐、协作、信誉系统提供基础数据

## 6. 用户画像

建议第一阶段只服务 3 类用户，不要过宽。

### 6.1 Web3 社群活跃者

特点：

- 常在 `Telegram`、`Discord`、`X` 中活跃
- 频繁加入新群、活动、项目社区
- 愿意尝试新工具，但耐心有限

需求：

- 快速展示身份
- 方便加人、找同类、找合作
- 希望有趣、轻量、能转发

### 6.2 Builder / 项目方成员

特点：

- 想认识开发者、设计师、运营、BD
- 需要在多个社群里传播自己的项目

需求：

- 个人或项目身份卡
- 社群活动入口
- 更高效的人才连接

### 6.3 KOL / 社区组织者 / 活动发起者

特点：

- 拥有流量和社群影响力
- 更看重传播工具和活动玩法

需求：

- 快速裂变
- 轻量 onboarding
- 可视化身份和成就展示

## 7. 核心产品逻辑

建议按照“身份 -> 连接 -> 传播 -> 沉淀”的逻辑设计。

### 7.1 身份层

用户第一次进入后，不应该先面对钱包和复杂表单，而是先完成一个极简身份初始化：

- 头像
- 昵称 / Handle
- 一句话介绍
- 标签，例如 `builder`、`designer`、`degen`、`founder`
- 所在生态或兴趣，例如 `Solana`、`Base`、`AI`、`NFT`
- 我正在找什么，例如 `合作`、`工作机会`、`组队`、`项目曝光`

这个页面本质上就是一个可传播的 `Web3 Bonjour Card`。

### 7.2 连接层

用户有了卡片之后，才进入互动。

可以设计 4 种最轻量的连接动作：

1. `Say Hi`
2. `Follow / Bookmark`
3. `Match by tags`
4. `Join same room / topic`

重点不是复杂聊天，而是“快速建立连接的理由”。

例如：

- 你们都在 `Base`
- 你们都在找 `AI x Crypto` 合作
- 你们都参加了同一个 Space、黑客松或群组

也就是说，连接不是随机社交，而是“带上下文的社交”。

### 7.3 传播层

这是最关键的一层。

产品必须天然适合在外部平台传播，所以所有核心对象都应该可链接、可分享、可嵌入：

- 个人卡片页
- 主题房间页
- 活动页
- 邀请页
- 匹配结果页

传播动作建议是：

- 分享到 `Telegram group`
- 分享到 `Discord channel`
- 分享到 `X`
- 贴到个人简介或 Link in Bio

分享后的落点必须非常轻：

- 点开就能看到对方身份
- 点开就能 `Say Hi`
- 点开就能创建自己的版本

### 7.4 沉淀层

当用户有了足够多的行为后，才逐步形成 DID。

沉淀内容包括：

- 身份资料
- 钱包绑定关系
- 社交账号绑定关系
- 参与过的群、活动、主题
- 收到和发出的连接
- 成就、徽章、声誉、贡献记录

这时 DID 不是“注册出来的”，而是“行为累积出来的”。

## 8. DID 应该怎么放

这是整个方案中最容易做重的地方。

### 8.1 建议原则

建议做一个**轻 DID**，不要上来做完整协议。

第一版可以把 DID 理解为：

**一个统一身份容器，用来把用户的多入口、多账号、多钱包、多社交行为关联起来。**

### 8.2 轻 DID 的组成

一个轻 DID 可以包含：

- 内部唯一 ID
- 用户自定义 Handle
- 钱包地址绑定
- `Telegram` 账号绑定
- `Discord` 账号绑定
- 可选的 `X` 账号绑定
- 标签、角色、兴趣、贡献记录
- 外部应用可读取的公开资料页

### 8.3 DID 的作用

它可以支持：

- 登录不同应用时复用身份
- 将社群行为映射为可验证档案
- 让合作方、活动方、其他 DApp 读取基础身份信息
- 后续扩展声誉系统、任务系统、证明系统

### 8.4 暂时不要做太重的部分

第一版不建议急着做：

- 复杂链上 DID 协议兼容
- 多链可验证凭证全覆盖
- 完整社交图谱上链
- 强依赖 gas 的身份写入机制

第一版更适合：

- 数据先以产品可用为目标
- 核心身份关系可签名或可验证
- 真正需要上链的部分再逐步抽象

## 9. Telegram / Discord 该怎么接

你提到依托 `Telegram` 或其他方式传播，这里建议从“入口”和“传播”两个角度拆开。

### 9.1 Telegram 更适合做入口和裂变

原因：

- Web3 用户高度集中
- 分享链接、拉人、进群路径自然
- Bot 和 Mini App 生态更适合冷启动

推荐玩法：

- `Telegram Bot` 作为入口
- 用户在 Bot 中生成自己的身份卡
- Bot 帮助分享卡片、邀请进入活动页或匹配页
- 后续如果需要，再做更完整的 Mini App 承载体验

### 9.2 Discord 更适合作为社群关系场景

原因：

- 频道和角色体系更适合社区运营
- 适合和活动、任务、频道分层结合

推荐玩法：

- Discord 用户通过频道链接进入产品页
- 把某个频道、活动、身份标签映射成一个“社交上下文”
- 用户在这个上下文里更容易建立连接

### 9.3 推荐顺序

建议优先级：

1. `Telegram 分享链接 + Bot 入口`
2. `独立 H5 / Web App 承接`
3. `Discord 社群嵌入和跳转`
4. `更完整的 Mini App 形态`

理由是先保证传播效率，再增加场景深度。

## 10. 最小可行产品 MVP

如果只做一个最小版本，我建议只做这 5 个模块。

### 10.1 身份卡创建

用户可以快速创建自己的卡片：

- 昵称
- 头像
- 标签
- 一句话介绍
- 我想认识谁

### 10.2 可分享个人页

用户获得一个可传播链接，别人打开就能看到：

- 你是谁
- 你在做什么
- 你想连接什么人
- 如何和你互动

### 10.3 Say Hi / 连接动作

用户点击一个按钮，就能向对方发起连接请求。

这个动作可以很轻：

- 留一句话
- 勾选自己的身份标签
- 选择目的，例如 `认识一下`、`合作`、`招募`、`加入社群`

### 10.4 场景化房间或活动页

例如：

- `ETH Beijing builders`
- `AI x Crypto founders`
- `Solana designers`
- `Hackathon afterparty`

用户不是单纯浏览，而是在某个明确主题里认识人。

### 10.5 基础身份绑定

第一版只做最有价值的绑定：

- 钱包
- Telegram
- Discord

这样就已经够形成一个初步 DID 骨架。

## 11. 推荐用户链路

下面是一条比较顺的用户路径。

### 11.1 初始传播

- 用户在 `Telegram` 或 `Discord` 看到某人的 Bonjour Card
- 点进去看到一个清晰、好玩的身份页
- 页面里有明显 CTA：`Create yours`

### 11.2 创建身份

- 用户只花 1 分钟生成自己的卡片
- 可以先不用连钱包，先完成轻注册
- 生成后立即得到分享链接

### 11.3 首次连接

- 用户把卡片分享到群或朋友圈
- 别人点开后给他 `Say Hi`
- 用户收到第一批互动

### 11.4 身份增强

- 用户绑定钱包和社交账号
- 补充更多标签和经历
- 加入某个活动页或主题房间

### 11.5 DID 成型

- 用户的身份和行为逐渐沉淀
- 其他应用开始调用或识别这个身份

## 12. 为什么别人会传播它

这是一个必须单独思考的问题。

用户传播一个产品，通常是因为下面四种动力之一：

- `表达自己`
- `获得社交反馈`
- `展示成就`
- `带来实际机会`

所以产品需要显式设计这些动机。

### 12.1 表达自己

让卡片足够好看、有趣、有风格、有 Web3 味道。

### 12.2 获得反馈

用户分享后能收到：

- Say Hi
- 关注
- 邀请
- 匹配结果

### 12.3 展示成就

卡片中可以展示：

- 链上经历
- 活动参与
- 社区身份
- 特殊徽章

### 12.4 带来机会

卡片不是纯展示，而是明确告诉别人：

- 我想找什么人
- 我能提供什么
- 我在哪些主题中开放连接

## 13. 产品差异化方向

如果你担心“这不就是一个名片工具吗”，那差异化要来自下面几个点。

### 13.1 它不是静态名片，而是连接引擎

重点不是展示，而是围绕标签、场景、活动和群组产生连接。

### 13.2 它不是纯 Web2 社交，而是可验证身份

用户可以逐步绑定钱包、证明参与、沉淀贡献。

### 13.3 它不是重社交网络，而是社群传播插件

它更像附着在 `Telegram`、`Discord`、`X` 上的身份连接层，而不是替代这些平台。

### 13.4 它不是先做协议，而是先做产品

先让人愿意用，再抽象 DID 和协议层。

## 14. 可扩展的第二阶段

第一版跑通后，再扩展下面这些能力。

### 14.1 声誉系统

- 被多少人连接
- 来自哪些圈层
- 是否有真实协作记录

### 14.2 可验证凭证

- 活动参与证明
- DAO 贡献证明
- 社群角色证明

### 14.3 社交图谱

- 你和谁经常在同一主题出现
- 你在哪些生态更活跃
- 你更容易和哪些类型的人建立连接

### 14.4 AI Agent 交互

未来 DID 可以不仅给应用看，也可以给 Agent 看：

- Agent 读取你的身份偏好
- Agent 帮你推荐人和群
- Agent 根据你的身份帮助完成应用内操作

这部分很适合你后面做“身份 + Agent + 应用交互”。

## 15. 变现思路

第一版不用急着变现，但可以提前想清楚。

### 15.1 面向用户

- 高级卡片样式
- 更强的身份展示组件
- 更高阶匹配能力

### 15.2 面向项目方 / 社群方

- 活动页和群成员 onboarding 工具
- 社群身份和匹配管理
- 招募、活动、AMA、黑客松场景模板

### 15.3 面向生态合作方

- DID / Profile API
- 推荐和匹配组件
- 活动或社区插件

## 16. 风险与难点

### 16.1 冷启动问题

如果没有足够多的人和场景，卡片会变成“发出去没人响应”。

解决思路：

- 先绑定具体活动或社群
- 用主题房间驱动连接
- 先做小圈层高密度使用

### 16.2 垃圾身份和机器人

Web3 产品很容易被 spam。

解决思路：

- 身份分层
- 绑定钱包 / Telegram / Discord
- 行为频控
- 社群邀请制或白名单场景

### 16.3 DID 过重

如果一开始就强调协议，会让用户无感。

解决思路：

- 用户先获得好用体验
- 身份系统在后台逐步增强

### 16.4 社交产品留存难

社交产品不是靠注册量，而是靠重复连接场景。

解决思路：

- 把产品和活动、社群、协作机会绑定
- 每次传播都有明确上下文

## 17. 最推荐的起步方向

如果让我帮你收敛成一个最适合起步的版本，我会建议：

### 方向定义

做一个：

**面向 Telegram/Discord 社群传播的 Web3 社交身份卡 DApp**

### 核心特征

- 轻注册
- 强传播
- 强标签化
- 弱聊天
- 先产品，后 DID

### 第一阶段只验证三件事

1. 用户愿不愿意创建自己的卡片
2. 用户愿不愿意把卡片分享出去
3. 分享后能不能产生真实连接

如果这三件事跑通，再去加：

- DID
- 声誉
- 凭证
- Agent 交互

## 18. 一个可直接对外描述的版本

你可以先这样描述这个项目：

> 一个为 Web3 社群打造的 Bonjour 式社交 DApp。  
> 用户可以在 Telegram、Discord 等场景中快速生成自己的身份卡片，表达自己是谁、在做什么、想认识谁，并通过轻量互动建立连接。  
> 随着用户绑定钱包、社交账号和参与行为，这个身份会逐步演化为一个可跨应用使用的轻 DID。

## 19. 下一步建议

如果继续往下推进，建议按这个顺序做文档而不是直接写代码：

1. 明确一句话定位
2. 选定第一批目标用户
3. 画出 MVP 页面和用户路径
4. 定义轻 DID 数据结构
5. 决定 Telegram / Discord 哪个先做入口
6. 再进入原型和开发
